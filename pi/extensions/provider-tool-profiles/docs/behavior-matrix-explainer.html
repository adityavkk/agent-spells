provider-tool-profiles, gear by gear

A coding model was trained against one specific tool harness. This extension hands it that harness's exact tools while running every call through Pi's own safety code. Here is how the machine does both at once.

TL;DR — The model sees the tool names and schemas it was trained on (Claude's Read, Codex's apply_patch, Gemini's read_file). The bytes run through Pi's execution primitives: a file-mutation queue, output truncation, abort checks, path containment. The contract comes vendored from Letta Code; the runtime is Pi. Two CI jobs watch the two halves so neither drifts without someone noticing. // one line to remember: provider-native contract, Pi-native runtime.

The one fact the rest follows from

Start here, because everything else is a consequence of it: vendoring a Letta schema records what a tool looks like to the model. It never runs Letta's code.

Under vendor/letta/ sit 25 JSON schemas and their descriptions, copied from letta-ai/letta-code at a pinned commit. A schema there is a contract: this tool is named Read, it takes file_path and an optional offset, it returns numbered lines. Letta already did the work of reverse-engineering and normalizing each provider's tool surface, so this extension copies that work instead of re-deriving it. But the copied file is a description of a shape, not an implementation. When the model calls Read, the bytes are read by a local function in tools/read-adapter.ts that uses Pi's file primitives. Letta is the blueprint. Pi is the factory.

Hold those two apart and the whole codebase reads cleanly. Let them blur and you will keep asking why a "vendored" tool isn't active, or why an active tool diverges from Letta's wording. The answer is always the same: the schema says what the model expects, the adapter decides what actually happens, and where they disagree the adapter wins and the disagreement is written down.

★
The split is enforced by a linter, not a habit. tools/pi-compat.ts:15 re-exports every Pi symbol the extension may touch. scripts/check-pi-compat.ts:161 fails CI if any other file imports a Pi package directly. :142 fails it if anyone reaches into dist/**, core/**, src/**, or modes/**. One file is the door; every other door is nailed shut.

What breaks if you skip it

A frontier model learns its tools the way it learns everything else: from examples during post-training. Claude saw a tool literally named Read that returns cat -n output counting from line 1. Gemini saw read_file counting from line 0 and returning plain text. Codex saw an apply_patch envelope grammar and a separate update_plan tool. The names, the argument shapes, and the output formatting are part of what the weights expect to find.

Pi ships its own tools: read, bash, edit, write, grep, find, ls. They work well and they are generic. Point a Claude-tuned model at the generic edit and it loses the sequential MultiEdit it reaches for, where the second edit operates on the text the first one produced. Point Codex at write and it never gets to emit the patch grammar it was trained to write. The model spends turns adapting to a tool it doesn't know instead of doing the work.

You could fix this by forking Pi and hand-porting each harness. That is a large maintenance bill and a security question: now you are running a third party's tool code. This extension takes the cheaper trade. It keeps the provider's tool surface and supplies its own tool body, so the model operates in its native dialect and every dangerous operation still goes through code Pi already trusts. The cost of that trade, paid in full later: every place the two harnesses genuinely disagree becomes a policy decision someone has to make and test by hand.

The gear train: model in, tools out

Five gears turn between picking a model and the model seeing its tools. Each one causes the next. Expand any gear for the call that drives it.

1 · Register all 21 tools at construction index.ts:28

When the extension loads it calls registerClaudeTools, registerCodexTools, and registerGeminiTools right away. Every tool definition is now known to Pi. Knowing a tool and showing it to the model are different things: registration happens once, here; activation happens later and changes with the model. A shared runtime object carries the session's read-history; a codexPlanState survives reloads.

2 · Resolve the real model behind the alias index.ts:36

On session_start and again on every model_select, syncTools() turns a possibly profile-backed alias into a concrete model with resolveProfileBackedModel(), reading the session branch so a mid-session switch counts. Two cases bail out before anything activates: a subagent child (PI_SUBAGENT_CHILD=1) and a disabled config. A nested agent never inherits a surprise tool swap from its parent.

3 · Read the model's fields to pick a profile provider-detect.ts:19

detectProviderToolProfile() checks the model's provider, id, and api strings against substring matchers from config. The env var PI_PROVIDER_TOOL_PROFILE can force a profile or shut detection off. Claude and Gemini are tested first; Codex carries an extra rule for OpenAI models whose id starts with gpt-. The matcher table is small enough to run yourself in the detection gear below.

4 · Swap the active set, remembering what to put back tool-activation.ts:43

buildProviderToolActivation() is a pure function, which is what makes the swap reversible. The first time a profile activates, it copies the current Pi core tools into previousCoreTools, then returns that profile's list plus any non-managed extension tools worth preserving. When the profile clears, it returns the saved core tools. This is the mechanism behind a property you would otherwise take for granted: switch models back and forth all day and you never get stranded with the wrong toolset.

5 · Tell the model which dialect is live tool-activation.ts:73

getProfilePromptAppendix() plus the before_agent_start hook (index.ts:89) add one sentence to the system prompt naming the active tools, for example "Provider tool profile: Codex. Use shell_command for shell work, apply_patch for file changes…". The append checks for itself first, so a reload does not stack three copies.

Who gets which tools, and who is held back

The manifest tracks far more tools than it turns on. The hinge fact from the top has a direct corollary here: a tool can be vendored and still be dark. Four statuses appear in tool-manifest.json. active means wired to a live adapter. vendored means the schema is tracked but no adapter runs it yet. blocked means activating it would need a Pi primitive that does not exist. ignored means out of scope, like Letta's own goal and memory tools.

ProfileActive tools (count)Held back, and why
Claude Read, Write, Edit, MultiEdit, Bash, Glob, Grep, LS (8) ignored BashOutput, KillBash: both need a background-shell registry Pi has not built.
Codex shell_command, apply_patch, update_plan, view_image (4) blocked exec_command, write_stdin · vendored shell, read_file, list_dir
Gemini run_shell_command, read_file, read_many_files, list_directory, glob, grep_search, search_file_content, replace, write_file (9) ignored WriteTodos: Pi already plans with update_plan.
⚠
Why exec_command and write_stdin sit blocked. They run a long-lived process you feed input to over time. That needs a session manager, stdin polling, cancellation, cleanup, and a decision about pseudo-terminals, none of which Pi exposes today. The manifest writes the reason into a decision field and leaves the tool dark. A half-built interactive-shell tool that loses your process is worse than a model that has to use shell_command instead.

What each tool actually does

Grouped by job. Each card gives three things: the contract the model sees, the Pi machinery underneath, and the one place they were made to disagree on purpose.

Read — Read · read_file · read_many_files tools/read-adapter.ts

Contract. Claude's Read returns cat -n numbered text counting from a 1-based offset. Gemini's read_file returns plain text counting from 0. Both stop at 2000 lines or 50 KB (policies.ts:24).

Pi underneath. Head-truncation with a "use offset N to continue" hint, byte-accurate caps, and base64 plus mime detection for png, jpg, gif, and webp.

The deliberate disagreement. Handing the model Pi's raw read output was tried and rejected as too blunt: it drops the line numbers and the offset base the model was trained on. So a thin adapter maps the offset in normalizedOffset (read-adapter.ts:65) and formats per profile. A binary file (it has a NUL byte) and a PDF, audio file, or notebook each return an explicit "unsupported" result instead of a screen of decoded garbage. Every successful text read writes a read-history entry. Run the offset split yourself in the read gear.

Edit — Edit · MultiEdit · replace tools/edit-adapter.ts:25

Contract. Exact literal string replacement. Claude's MultiEdit is sequential: edit 2 runs against the text edit 1 produced, and the batch is atomic. replace_all and expected_replacements are honored, and a count that does not match throws (shared.ts:96).

Pi underneath. The whole read-modify-write runs inside withFileMutationQueue(), keyed by path, so two edits to one file take turns. The abort signal is checked before and after every await.

The deliberate disagreement. Pi's native edit matches all of its edits against the original text. That is the opposite of sequential, so this code does not delegate to it. applyExactEditsToText() folds the edits left to right over the text as it changes, which is the behavior Claude expects.

Shell — Bash · shell_command · run_shell_command tools/shell-adapter.ts

Contract. A nonzero exit code is a normal result the model reads and reacts to, not a failure. A timeout or an abort is a failure. The invocation matches each harness (shell-adapter.ts:99): Claude runs bash -lc, Gemini runs bash -c, Codex runs bash -lc by default but bash -c when the call sets login: false.

Pi underneath. Execution rides ExtensionAPI.exec. Output is tail-truncated, and an overflow writes the full output to a temp file whose path comes back in the result. Timeouts clamp to a ceiling.

The safety line. A field that changes the security posture is refused, not ignored. Codex sandbox_permissions: "require_escalated" returns an explicit unsupported result before exec (shell-adapter.ts:61). So do justification, a non-empty prefix_rule, and Claude's run_in_background (:133). Quietly accepting an escalation you cannot enforce is the one outcome worse than refusing it.

Search and list — Glob · Grep · glob · grep_search · LS · list_directory tools/search-adapter.ts, tools/list-adapter.ts

Contract. Glob returns newest file first, capped at Letta's vendored 2000. Grep keeps the provider output modes (content, files_with_matches, count) and the search_file_content alias for grep_search.

Pi underneath. One ripgrep call, stable path sorting, result caps, truncation notices.

The deliberate disagreement. A local ignore engine in tools/ignore-policy.ts reads anchored rules, directory-only rules, escaped # and !, ordered negations, and scoped ignore files. Gemini honors .geminiignore for glob, grep, and list, and .gitignore for list unless told not to; respect_git_ignore: false becomes ripgrep --no-ignore-vcs. It is not a full reimplementation of Git's ignore engine, and the code says so.

Codex apply_patch — the one bespoke engine tools/apply-patch.ts:313

Contract. Codex's *** Begin Patch … *** End Patch envelope, with Add, Update, Delete, and Move operations. Paths are relative, never absolute.

Pi underneath. A per-file mutation queue with a stable lock order so a multi-file patch cannot deadlock against itself.

The deliberate disagreement. Pi has no equivalent, so this is custom code, built in three phases: parse, then a read-only preflight that computes every new file in memory, then commit. Disk stays untouched until the entire patch has parsed, passed path policy, and applied cleanly in memory. It keeps in-memory snapshots and rolls back on a failed commit, and it does not claim to be crash-safe. Step the phases in the apply_patch gear.

Codex update_plan and view_image tools/plan-state.ts, tools/read-adapter.ts:155

update_plan. It allows at most one step marked in_progress, writes the latest plan as a provider-tool-profiles.codex.plan.v1 session entry through Pi's appendEntry(), and reloads it on session_start. A reload or resume keeps the plan the model was working from.

view_image. It shares the image read path but keeps Codex's own path rule, resolveCodexImagePath (path.ts:198): absolute or relative, read-only, no cwd containment. It is the one file tool with no containment, because it cannot write and screenshots usually live outside the repo.

The two watchdogs

    
The contract and the runtime drift for different reasons, so each gets its own watchdog. Letta can change a schema upstream. Pi can change a public API upstream. Neither job can see the other's failure, and that is the point.

      What Letta drift actually compares scripts/check-letta-drift.ts
      

        
It reads tool-manifest.json at its pinned commit 2eca6e1, then for each tracked tool fetches the upstream schemas/*.json and descriptions/*.md from raw.githubusercontent.com and diffs them against the vendored copies. It also walks the upstream git tree to find tools that exist there but not in the manifest, and diffs default-toolsets.json. The report is non-failing on purpose: a contract change becomes a tracked issue a human reads, not a red build that blocks unrelated work. Today it reports changed=0 newSchemas=0 toolsetDiffs=0.

      

    

    

      What Pi compatibility actually runs scripts/check-pi-compat.ts
      

        
Four checks. First, the full provider-tool test suite. Second, a static scan of every .ts file's imports: a Pi package imported outside tools/pi-compat.ts is a boundary violation (:161), and any dist/, core/, src/, or modes/ path is a private-import failure (:142). Third, a smoke test that the 16 required exports still exist and behave, down to formatSize(1024) === "1.0KB". Fourth, the same suite against the latest published Pi in a temp dir, so an upstream break shows up before you upgrade rather than after.

      

    


    
★
The rule that ties the watchdogs together. A Letta-driven tool change ships only when both jobs are green for that capability. If Letta wants a tool whose behavior needs a Pi primitive that does not exist, you vendor it and mark it blocked rather than ship behavior Pi cannot back. That rule is the whole reason exec_command and write_stdin are dark right now: the contract is ready, the runtime is not, so the tool waits.

Glossary

    

      
Offset base

      
The line number a read tool treats as the first line. Claude counts from 1, Gemini from 0. The adapter maps both onto the same 0-based array slice.

      
Core tools

      
Pi's own generic tools: read, bash, edit, write, grep, find, ls. Copied before a profile activates so they can be put back.

      
Read-history

      
A per-session map of canonical path to {mtime, size, sha256}. Lets an edit report whether the file is fresh, stale, or never-read. Recorded today, enforced never.

      
Preflight

      
apply_patch's read-only phase: parse, resolve paths, and compute every new file in memory. No disk write happens until it finishes clean.

      
Compat boundary

      
tools/pi-compat.ts, the only file allowed to import a Pi package. CI fails any other file that imports Pi directly.
